## Project purpose

This is a **static HTML/CSS/JS brew-day web app** for real use during homebrew sessions.
It is not a generic brewery website.
It should feel like a warm, polished, mobile-friendly brew-day cockpit with Funcleson personality.

Core goals:
- useful during actual brew days
- works well on iPhone
- supports home-screen install behavior
- uses Firebase Auth + Firestore for persistence when hosted
- still has a usable local-only mode for development/testing
- deploys by drag/drop to Netlify as a static site

## Current user workflow assumptions

The user’s real workflow is now:
1. create a recipe
2. save the recipe template
3. load the recipe into the app on brew day
4. use Brew tab and Timers during the live session
5. leave notes / gravity / checklist updates during the day
6. package/store later
7. archive finished batches

This means:
- recipe loading and recipe reuse are critical
- Brew should be based on a loaded recipe/batch, not a blank batch-first workflow
- archive loop-back and rebrew behavior are high-value

## Important architectural change

The project was intentionally split out of single-file mode because the old giant file had the classic single-file failure mode:
- deleting a DOM node but forgetting a renderer still targets it
- changing one ID and silently breaking a different tab
- mixing styling, UI rendering, and brewing math in one place
- one syntax issue breaking the whole app

### Current file roles

#### `index.html`
Contains layout and containers.
Should stay relatively clean.

#### `styles.css`
Contains styling and responsive behavior.

#### `app.js`
Contains state, rendering, event wiring, persistence, workflow logic, tab behavior, and BrewMate behavior.

#### `brew-logic.js`
Contains brewing formulas and reusable brewing math.
Keep brewing calculations here instead of embedding formulas throughout UI code.

## Current tab intent

### Dash
Dashboard should be about the **active brew only**, not vanity/archive metrics.
Surface live-useful data only:

- active timers
- current batch pulse
- what needs attention now
- gravity snapshot
- finish/archive readiness

### Recipes
Recipes are central.
The user cares a lot about saved recipes as durable value.

Recipes should support:
- creating recipe templates
- saving recipes
- selecting a recipe
- loading a recipe into the active batch
- BIAB-first planning inputs

Important recipe changes already made:
- mash-guidelines-style content was removed because this app is now BIAB-focused
- BIAB water planning was moved into the recipe workflow
- duplicate strike temperature calculators were consolidated because mismatched math was eroding trust

### Brew
This is the operational tab once a brew is underway.
It should contain:
- loaded batch summary
- running batch notes
- beer-only brew-day checklist
- BIAB brew plan
- gravity logging
- quick operational actions

Important:
- the brew-day checklist should be **beer only**, not mead
- the BIAB brew plan in Brew was reformatted into **clean rows**, not one long unreadable sentence
- recipe BIAB defaults now push forward into Brew so the workflow is continuous

### Timers
Timers are high-value and should feel fast.

Preferred emphasis:
- active timer dock
- live boil snapshot
- current bench snapshot
- timer cards and schedule controls below

Important timer bug already fixed:
- timer preset fields were previously being overwritten while typing because live render cycles kept stomping on the input
- current behavior should allow editing preset values normally while focused, with updates applied on blur/change/Enter
- do not accidentally reintroduce the “can’t edit timer numbers” bug

### Storage

### Archive

Important:

### BrewMate
This is the newest major feature and should be treated as important, not disposable.

BrewMate is **not** supposed to be a fake chatbot.
It is a structured **creative brewing mentor** / concept lab.

Its current purpose:
- brainstorm beer concepts
- mentor through style/flavor/adjunct thinking
- teach what goes with what and why
- identify likely conflicts / clutter / muddy design choices
- help the user design a concept first, then turn it into a recipe draft

BrewMate currently includes concept inputs like:
- beer idea name
- style / anchor beer
- inspiration / vibe
- drinking experience
- batch size
- target ABV / IBU
- bitterness frame
- body
- finish
- yeast expression
- drinkability target
- experimental level
- learning focus
- must-have flavors
- avoid / keep out
- adjuncts / signature ingredients
- constraints / concerns

BrewMate currently outputs boards/cards such as:
- pairings that support this beer
- ingredient coach
- concept north star
- recipe architecture
- conflict check
- why this works

Important BrewMate workflow buttons/features already added:
- pull selected recipe
- pull live batch
- send starter draft to Recipes
- copy deep-dive prompt
- open ChatGPT
- clear inputs

Important BrewMate intent:
- it should feel like a brewer’s brainstorming mentor
- it should teach pairings, structure, balance, and tradeoffs
- it should help decide on flavor profile / style / adjunct logic before recipe drafting
- it should remain usable offline / locally, not depend on a live backend AI service

Knowledge intent for BrewMate:
- it should be grounded in real brewing/style/sensory/ingredient logic
- inspiration should reflect tried-and-true brewing knowledge and culinary pairing sense
- however, it is still a local rules/heuristics engine, not a live all-knowing model
- avoid pretending it contains every expert opinion; instead keep it practical, disciplined, and useful
